## Качество — выявленные эффекты скилла

### Что работало хорошо

1. **Дисциплина параллельных батчей** — все 9 Read'ов Слоя 3 в одном turn'е, WebSearch в одном батче. Без скилла шло последовательно.
2. **Объём в target 3000–4500 слов** — соблюдён в 21 из 26 случаев. У 5 досье (Елена, Марат Бусурин, Юнга, Антоша, Смугляночка) был чуть меньший объём 2693–3034 слов.
3. **Не-триплоидность Папировки** — закреплена в скилле, ни одно досье не повторило мою исходную ошибку.
4. **Корректная атрибуция Жебровского** (Лисавенко, не Бакчар) — выполнено во всех релевантных досье.
5. **Качество расхождений с darwinshop** — в среднем 4–6 пунктов на досье, включая критические (Папировка ≠ Белый налив, Налив розовый ≠ Папировка, неуказанный подвой, опечатка «Басурин» вместо «Бусурин», ошибка с Группой допуска).

### Что нужно доработать в скилле

1. **Альтернативы из Группы 17.** Скилл говорит «из ДРУГОЙ группы Б1», что для Группы 1 сортов выдаёт парадоксальный результат (понизить зимостойкость клиенту). Правило должно быть: **«альтернативы лучше или равны основному сорту для Томска»**. Для Группы 1 → партнёры той же группы; для Группы 3 → варианты из Группы 1.
2. **Сорт «на грани» — двойная классификация.** Несколько сортов попали в Группу 1.1 формально, но фактически на грани (Алтынай +0…+1, Сувенир Алтая 0…+3, Толунай −5…0, Стройное −1…−3). Нужно явное правило «нижняя граница группы → возвышенный участок обязателен».
3. **Папировка ≠ Белый налив как находка раздела 1.** Только агенты с глубокой проработкой (Opus xhigh без скилла) её делают отдельным пунктом. Скилл нужно дополнить требованием «искать синонимы сорта в Госреестре и явно отделять их».
4. **Заведомо запрещённые препараты в календаре обработок.** Скилл должен явно указывать что в Слое 3 уже есть актуальная база (ДНОК, Нитрафен, Калипсо запрещены/отозваны) — не надо переисследовать.

## Рекомендации для масштабирования на полные 600 сортов

### Оптимальная схема (с подпиской Claude Code)

1. **Pre-fetch фаза** — написать Python-парсер ВНИИСПК + Госреестра, кеширующий первоисточник в JSON для каждого сорта. Один раз, прогон ~2–4 часа, обнуляет потребность агентов в WebSearch для базовых фактов.
2. **Sonnet + skill для bulk-generation** — 600 досье × $1.09 = **$654**, время ~6.3 ч × 10 параллельных = ~38 ч wall-clock волнами.
3. **Opus верификация** — короткий Opus-агент проходится по факт-чеку 1–2 турна, ~$0.30/досье × 600 = **$180**.
4. **Итого: ~$830 за 600 досье с гарантией качества Opus.**

### Альтернатива — только Opus

26 × $2.44 = $63 проект исполнен. Линейная проекция: 600 × $2.44 ≈ **$1,465**. Может быть выше из-за рост сложности (Wave 5+6 показала $2.78 в среднем для менее документированных сортов). Реалистично — **$1,500–1,800**.

### Если переходить на SDK API напрямую (отдельный биллинг)

- Batches API даёт 50% скидку → проекция падает до **$700–900**.
- `cache_control ttl: "1h"` устранит rewriting → ещё −20%.
- `effort: "medium"` на Opus → −30% при сохранимом качестве.

## Что осталось за рамками пилота

- **3 земляники** в каталоге (Азия, Альба, Альбион) — не обработаны, методика для земляники отличается (нет понятий штамба/стланца).
- **Декоративная яблоня Роялти** (ошибочно в категории «Черёмуха» в магазине) — отдельный случай, нужен декоративный шаблон.
- **Остальные ~570 сортов** каталога darwinshop (всего на сайте 630 товаров) — пилот ограничился яблонями.
- **Векторная БД и интеграция с чат-ботом** — отдельная задача, после полного RAG-собрания.
- **CI-обновление** — досье генерируются один раз; механизм обновления при изменении наличия в darwinshop требует отдельной разработки.
